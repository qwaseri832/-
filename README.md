## 📖 О проекте

**In-Memory Message Broker** — простой брокер сообщений на Go, работающий по протоколу HTTP. Это легковесная альтернатива RabbitMQ или Redis для случаев, когда не нужна персистентность и репликация.

Сервис поддерживает:
- Отправку сообщений в очереди (`PUT`)
- Получение сообщений с ожиданием (`GET` с `timeout`)
- FIFO (первым пришёл — первым ушёл)
- Long polling для блокирующего чтения

## 🔑 Как это работает

### Структуры

```go
type queue struct {
    msgs    []string    // буфер сообщений
    waiters []chan string // ожидающие получатели (FIFO)
}

type broker struct {
    mu     sync.Mutex
    queues map[string]*queue // очередь по имени
}
```

### Логика работы

1. **`PUT`** — кладёт сообщение:
   - Если есть ожидающий получатель — отдаёт ему сразу
   - Если нет — сохраняет в буфер

2. **`GET`** — забирает сообщение:
   - Если есть в буфере — отдаёт первое
   - Если пусто и `timeout > 0` — встаёт в очередь ожидания
   - Если `timeout = 0` или истёк — возвращает `404`

## 📡 API

### `PUT /{queue_name}?v={message}`

Отправить сообщение в очередь.

**Пример:**

```bash
curl -X PUT "http://localhost:8080/notifications?v=Hello%20World"
```

**Ответ:**

- `200 OK` — сообщение отправлено
- `400 Bad Request` — параметр `v` отсутствует

---

### `GET /{queue_name}?timeout={seconds}`

Забрать сообщение из очереди.

**Пример:**

```bash
curl "http://localhost:8080/notifications?timeout=30"
```

**Ответ:**

- `200 OK` — тело ответа содержит сообщение
- `404 Not Found` — сообщения нет (или таймаут истёк)

---

## 🔄 Пример работы

```bash
# Терминал 1 — получатель ждёт сообщение (таймаут 60 сек)
curl "http://localhost:8080/notifications?timeout=60"

# Терминал 2 — отправитель кладёт сообщение
curl -X PUT "http://localhost:8080/notifications?v=Order%20processed"

# Терминал 1 — получит "Order processed"
```

## 🔑 Ключевые особенности

| Особенность | Описание |
|-------------|----------|
| **In-Memory** | Все данные хранятся в оперативной памяти |
| **FIFO** | Сообщения обрабатываются в порядке поступления |
| **Long Polling** | Получатель может ждать сообщение до таймаута |
| **Потокобезопасность** | Используется `sync.Mutex` для защиты от гонок |
| **Простота** | Один файл, нет зависимостей |

## ⚠️ Ограничения

- ❌ Нет персистентности — при перезапуске все сообщения теряются
- ❌ Нет репликации — только один экземпляр
- ❌ Нет подтверждений (ack) — сообщение удаляется сразу после получения
- ❌ Нет TTL (время жизни сообщений)
- ❌ Нет приоритетов
